Fix ITMS-90078:
Missing Export Compliance Key

When this key is missing, App Store Connect asks export compliance questions on each upload because your Info.plist doesn't declare ITSAppUsesNonExemptEncryption. Add the key to avoid repeated compliance prompts.

‚ö° Quick Answer

ITMS-90078: Missing Export Compliance Key means your app's Info.plist doesn't declare ITSAppUsesNonExemptEncryption. Without this key, App Store Connect prompts you with export compliance questions on every single upload, adding friction and delay to your release process. Fix: Add <key>ITSAppUsesNonExemptEncryption</key><false/> to your Info.plist. If your app only uses HTTPS for networking (which covers 95% of apps), false is the correct value.

What Triggers ITMS-90078?

U.S. export regulations (administered by the Bureau of Industry and Security) require apps distributed through the App Store to declare whether they use encryption that goes beyond standard exemptions. Apple implements this via the ITSAppUsesNonExemptEncryption key in Info.plist.

If this key is missing, App Store Connect can't determine your compliance status automatically, so it shows you a manual questionnaire after every upload. This isn't a hard rejection like ITMS-90683, but it's a workflow killer ‚Äî especially for teams doing frequent builds or CI/CD deployments.

What If You Actually Use Non-Exempt Encryption?

If you set ITSAppUsesNonExemptEncryption to true, you'll also need to:

  1. File a SNAP/CCATS classification or Encryption Registration Number (ERN) with the U.S. Bureau of Industry and Security (BIS)
  2. Send a copy of the classification to Apple via Apple Developer Contact
  3. Add the ITSEncryptionExportComplianceCode key to your Info.plist with your compliance document number

Most apps ‚Äî including apps using HTTPS, CryptoKit, or standard cloud SDKs ‚Äî are exempt and should use false. See Apple's documentation for the full policy.

Frequently Asked Questions

Does using HTTPS count as "non-exempt encryption"?
No. Standard HTTPS via URLSession, Alamofire, or any standard HTTP library is exempt under U.S. export regulations (BIS EAR ¬ß740.17). Set ITSAppUsesNonExemptEncryption to false.
What about Firebase, AWS, or other cloud SDKs?
Cloud SDKs like Firebase, AWS Amplify, and Supabase use standard HTTPS for their network communication. These are exempt. You can safely set the key to false when using these SDKs for standard features (auth, database, storage, analytics).
What about CryptoKit or CommonCrypto?
If you use CryptoKit or CommonCrypto for on-device operations (hashing, signing, local encryption), these are generally considered exempt as they use Apple's built-in encryption. However, if you're using them to implement a custom encrypted communication protocol, you may need to set the key to true.
Will this fix work for Fastlane / CI/CD uploads too?
Yes. When the ITSAppUsesNonExemptEncryption key is in your Info.plist, both xcrun altool and Fastlane's deliver / pilot will skip the compliance prompt. This is essential for fully automated deployment pipelines.
Is this the same as the "Export Compliance" checkbox in App Store Connect?
Yes ‚Äî the Info.plist key is the binary-level declaration that replaces the manual checkbox in App Store Connect. When the key is present, App Store Connect reads it automatically and skips the dialog. It's the same declaration, just embedded in your app binary instead of answered manually.
